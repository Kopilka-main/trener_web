# Перенос в Фазу 2 (из ревью Фазы 1)

Зафиксировано по итогам code-review фундамента (Фаза 1). Не блокеры Фазы 1 — учесть при планировании Фазы 2.

## Архитектурные

1. **Zod-сериализация ответов через type-provider.** Сейчас health-роут возвращает объект по типу `HealthResponse` без рантайм-сериализации схемой. До появления доменных модулей завести паттерн валидации/сериализации ответов (например `fastify-type-provider-zod` или `schema.response`), чтобы доменные модули копировали его, а не подход «только тип». Соответствует инварианту в `AGENTS.md` («сериализация выхода — только Zod-схемами»).

2. **Ветка нативной Fastify-валидации в error-handler.** `apps/api/src/plugins/error-handler.ts` обрабатывает `AppError` и `ZodError`; нативные fastify-валидационные ошибки (`error.validation`) сейчас уходят в общий 500-fallback. Добавить ветку `error.validation` → 400, когда у роутов появятся schema-валидации запросов.

## Инфраструктурные

3. **Исключить тестовые файлы из прод-сборки.** `tsc -b` сейчас включает `*.test.ts`/`*.itest.ts` (это даёт типизацию тестов — ценно), но они попадают в `dist` и в Docker-образ. Решение без потери типизации тестов: отдельный `tsconfig.build.json` (extends основной + `exclude` тестов) для скрипта `build` и Dockerfile, тогда как `typecheck` (`tsc -b`) продолжает покрывать тесты. Применить до прод-деплоя (Фаза 7), но удобнее заложить в начале Фазы 2.
