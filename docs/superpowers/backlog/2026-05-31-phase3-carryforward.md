# Перенос в Фазу 3 (из ревью Фазы 2)

Зафиксировано по итогам code-review auth-фундамента (Фаза 2). Не блокеры Фазы 2 — учесть при планировании Фазы 3 (доменное ядро: `clients` + `trainer_clients`).

## Безопасность

1. **Регистро-независимый unique-индекс email.** Сейчас уникальность email держится на нормализации в Zod-контракте (инвариант задокументирован в `schema.ts`). При появлении seed/импорта/админки перейти на `uniqueIndex(...).on(lower(email))` или `citext`, чтобы БД сама защищала от дублей `A@b.co`/`a@b.co`.
2. **Тайминговая анти-энумерация login.** При отсутствии тренера `verifyPassword` не вызывается → ответ быстрее, чем при существующем email (argon2 дорогой). Тело/статус одинаковы (401), митигируется rate-limit (20/мин). При необходимости добавить «dummy verify» для выравнивания времени.
3. **CSRF при мутациях.** `sameSite: 'lax'` достаточно для текущих POST-эндпоинтов auth. Когда появятся cookie-аутентифицированные мутирующие доменные роуты, решить: `sameSite: 'strict'` для чувствительных операций либо CSRF-токен/double-submit (особенно при кросс-сабдомен фронтенде). Зафиксировать допущение явно.

## Качество тестов/кода

4. **Типизировать мок-фабрику repo вместо `as never`** (`auth.service.test.ts`): фабрика как `(): AuthRepo` с полноценными заглушками — тогда дрейф контракта repo ловится компиляцией, а не в рантайме. Применить как эталон для service-тестов Фазы 3.
5. **Изоляция состояния между repo-itest.** Сейчас в `auth.repo.itest.ts` два `it` делят состояние (t1/s1). Для Фазы 3 (много repo-тестов) — per-test cleanup, а не общая цепочка.

## Архитектура (composition root)

6. **Per-module register-хелпер.** Вся DI-проводка инлайн в `buildApp`. Когда доменных модулей станет 2–3, извлечь `registerClientsModule(app, deps)` и т.п., чтобы composition root оставался читаемым. `AppDeps` может расти без слома сигнатуры.

## Прямо в объёме Фазы 3 (из spec)

7. **Доменное ядро:** `clients` (без `user_id`) + `trainer_clients` (M:N) — эталон scope-изоляции по `trainerId` в repo; `requireClientAccess(clientId)` (проверка связи через `trainer_clients`, отказ → 404). Использовать `requireAuth` (уже готов как seam) как preHandler доменных роутов.
