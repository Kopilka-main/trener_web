# Дизайн: поддержка нескольких тренеров у клиента

Дата: 2026-07-16
Статус: утверждён (дизайн), к плану реализации

## Проблема

Один человек = один клиентский аккаунт (`client_accounts`, идентификатор
`clientAccountId`). Разные тренеры заводят каждый свою карточку в таблице
`clients`, но с одним и тем же `accountId` — то есть **несколько активных
привязок на аккаунт технически уже возможны**, и флоу привязки (`claim`) их
создаёт additively (проверка `findByAccountId(trainerId, code)` — по паре
тренер+аккаунт, поэтому второй тренер получает новую карточку).

Но весь client-facing API упирается в единый резолвер
`findScopeByAccountId` (`apps/api/src/modules/client-auth/client-auth.repo.ts`):

```
.orderBy(asc(createdAt), asc(trainerId)).limit(1)   // берёт самого раннего тренера
```

Он возвращает **одну** привязку `{ trainerId, clientId }`, и она через
`makeClientScope` (`apps/api/src/core/client-scope.ts`) подставляется во **все**
client-app модули: `chat, calendar, workouts, packages, measurements,
progress-photos, exercises, templates, trainer`.

Следствие: если у клиента 2+ тренера, приложение молча показывает данные только
самого первого — везде (чат, календарь, тренировки, пакеты, прогресс), а не
только в чате. Эндпоинта «список моих тренеров» нет; `GET /api/client/trainer`
отдаёт того же единственного.

## Продуктовые решения

1. **Модель — глобальный «активный тренер».** Клиент выбирает тренера, и весь
   клиентский раздел (чат, календарь, тренировки, прогресс, пакеты) показывает
   данные только этого тренера. Переключение меняет весь контекст — как отдельный
   «кабинет» под каждого тренера. Агрегации/слияния между тренерами нет.
2. **Вход — экран-контакты + переключатель.** При запуске: 1 тренер → сразу на
   главную (текущий UX без изменений); 2+ → сперва экран контактов (список
   тренеров с аватаром/именем/непрочитанным), выбрал → вошёл в его кабинет. Смена
   тренера — через переключатель (возврат к контактам).
3. **Пуши и непрочитанное — со всех тренеров.** Пуши приходят от всех тренеров
   независимо от активного (устройство/подписка привязаны к аккаунту, а не к
   тренеру). Тап по пушу переключает активного тренера на того, кто написал, и
   открывает нужный экран. В контактах — бейдж непрочитанного на каждом тренере;
   общий счётчик — сумма по всем.

## Архитектура (обзор)

Заменяем «зашитый первый тренер» на «выбранный тренер»:

- **Бэкенд:** скоуп-мидлвара начинает уважать переданную клиентом привязку;
  добавляется эндпоинт списка тренеров; пуши клиенту несут идентификатор
  тренера/привязки.
- **Мобильное приложение (client):** хранит активного тренера, шлёт его в
  заголовке каждого запроса, а при переключении перезагружает все разделы.

Выбранный технический подход — **заголовок скоупа** (Вариант 1 из
brainstorm): минимально инвазивно (одна правка ядра скоупа), пути роутов не
меняются, обратная совместимость со старыми версиями приложения, stateless.
Отвергнутые: активный тренер в серверной сессии (stateful, гонки между
устройствами) и `clientId` в пути каждого роута (правит все сигнатуры и все
мобильные вызовы).

## Бэкенд

### B1. `GET /api/client/trainers` (новый)

Список всех активных привязок аккаунта:

```
200 → { trainers: [{ clientId, trainerId, trainerName, avatarFileId, unread, lastMessageAt }] }
```

Репозиторий: `trainer_clients × clients × trainers`, где `clients.accountId =
<аккаунт из сессии>` и `trainer_clients.status = 'active'`; join непрочитанного
и времени последнего сообщения по каждой привязке; сортировка по `lastMessageAt
desc` (привязки без переписки — в конце по `createdAt`). Требует `requireClient`
(есть `clientAccountId`), но НЕ использует `scope` (он как раз про одного).

### B2. Правка `makeClientScope` (ядро, одна точка)

`apps/api/src/core/client-scope.ts`:

- заголовок `X-Client-Scope: <clientId>` присутствует → резолвим через новый
  `findScopeByAccountAndClient(accountId, clientId)`, который возвращает привязку,
  только если `clients.id = clientId`, `clients.accountId = accountId` и
  `trainer_clients.status = 'active'`; используем её;
- заголовка нет → текущее поведение `findScopeByAccountId` (первый тренер) —
  **обратная совместимость** со старыми версиями приложения;
- заголовок есть, но привязка не принадлежит аккаунту (резолвер вернул null) →
  `409 NOT_LINKED` (клиент перевыберет тренера).

Новый метод резолвера добавляется в `client-auth.repo.ts` и пробрасывается тем же
способом, что и текущий `resolveScope` (через `client-auth.service` в `app.ts`).
`ResolveScope` расширяется до формы, принимающей опциональный целевой `clientId`,
либо добавляется параллельный `resolveScopeFor` — детали в плане.

### B3. Пуши несут тренера

Сейчас клиенту уходит `url: '/chat'` / `'/notifications'` без идентификатора
(`apps/api/src/modules/chat/chat.service.ts` и др.). Добавляем скоуп-параметр:

- новое сообщение → `url: '/chat?scope=<clientId>'`;
- задача/замер → `'/notifications?scope=<clientId>'`;
- занятие на согласование → `'/calendar?scope=<clientId>'`.

Правки во всех местах, где формируется пуш клиенту (chat, sessions/calendar,
measurement-tasks). `<clientId>` — это карточка клиента у соответствующего
тренера (та же, что в скоупе), по ней мобилка восстановит активного тренера.

### B4. Общий счётчик непрочитанного

Существующий `GET /api/client/chat/unread` остаётся без изменений (он про
активного тренера, используется внутри контекста тренера). Сумму по всем
тренерам мобильное приложение считает из ответа `GET /api/client/trainers` (поле
`unread` на каждой привязке). Отдельный «total»-эндпоинт не нужен (YAGNI).

## Мобильное приложение (client)

### M1. Состояние

- **`activeTrainerProvider`** — хранит выбранную привязку
  `{ clientId, trainerId, name, avatarFileId }`; персистится в `LocalJsonStore`
  (переживает перезапуск). `null` = не выбран.
- **`clientTrainersProvider`** (FutureProvider) — `GET /api/client/trainers`.

### M2. Заголовок скоупа в ApiClient

В core (`packages/core/lib/src/api/api_client.dart`) добавляем опциональный
`scopeProvider` (по образцу `tokenProvider`): Dio-интерсептор ставит
`options.headers['X-Client-Scope'] = <clientId>`, если активный тренер выбран.
Клиентское приложение передаёт `scopeProvider`, читающий `activeTrainerProvider`;
тренерское приложение — не передаёт (заголовок не появляется).

### M3. Гейт роутера (вход в авторизованную зону)

Опираясь на `clientTrainersProvider`:

- 0 тренеров → текущий онбординг «подключись к тренеру» (connect/QR);
- **1 тренер → авто-выбор этой привязки, сразу на главную — текущий UX без
  изменений;**
- 2+ и активный не выбран → **экран контактов**; после выбора — ставим активного
  и на главную.

### M4. Экран контактов (новый)

Список тренеров: аватар, имя, бейдж непрочитанного, время последнего сообщения.
Тап → `activeTrainerProvider = привязка`, переход на главную. Возврат к экрану —
через **переключатель** в шапке главной (аватар + имя активного тренера →
«сменить тренера»).

### M5. Переключение = инвалидация

Смена активного тренера сбрасывает все клиентские data-провайдеры (чат,
календарь, тренировки, главная, пакеты, замеры, фото, уведомления, непрочитанное),
чтобы экраны перезагрузились в новом скоупе. Переиспользуем существующий паттерн
`resetUserScopedData` / `UserScopeObserver` (смена скоупа ≈ смена пользователя).

### M6. Пуши

- **Тап по пушу**: читаем `?scope=<clientId>` из url; ставим активного тренера на
  эту привязку (данные берём из `clientTrainersProvider`), затем навигация на
  целевой экран (`/chat`, `/notifications`, `/calendar`). Если `scope` нет среди
  привязок (тренер отвязал) → на контакты.
- **Пуш в форграунде**: как сейчас инвалидируем данные + обновляем список
  тренеров (для бейджей непрочитанного).

## Граничные случаи

- **Старая версия приложения** (без заголовка) → бэк падает в фолбэк «первый
  тренер», приложение продолжает работать.
- **Тренер отвязал клиента**: список тренеров сократился; если пропал активный →
  уходим на контакты (или авто-выбор, если остался один).
- **Клиент отвязался сам**: существующий флоу (`detachAccountFromClient`).
- **Заголовок с чужим/несуществующим clientId** → `409 NOT_LINKED`.

## Тестирование

- **Бэкенд (unit):** резолвер скоупа (владеет / не владеет / без заголовка);
  репозиторий списка тренеров (несколько привязок, непрочитанное, сортировка);
  формирование пуш-url со `scope`.
- **Бэкенд (itest, только против `trener_test`):** `GET /api/client/trainers` с
  несколькими привязками; скоупленные запросы client-app с заголовком
  `X-Client-Scope` (правильный тренер / 409 на чужой / фолбэк без заголовка).
- **Мобилка:** `flutter analyze`; ручной смоук — привязать 2 тренеров,
  переключаться, проверить, что каждый раздел (чат, календарь, тренировки,
  прогресс, пакеты) отдаёт данные выбранного тренера; проверить пуш-переход с
  переключением активного.

## Вне scope (не делаем сейчас)

- Слияние/агрегация данных разных тренеров в один список (осознанно отвергнуто в
  пользу модели «активный тренер»).
- Отдельный «total unread» эндпоинт (сумма считается на клиенте).
- Изменения во флоу привязки тренера (он уже additive и работает).
