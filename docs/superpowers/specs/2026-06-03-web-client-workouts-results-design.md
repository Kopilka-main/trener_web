# Клиентское приложение — раздел «Тренировки» (результаты), v1

Дата: 2026-06-03
Статус: дизайн на ревью
Ветка: `feat-web-design` (работаем на ней; фундамент клиентского приложения уже влит)

## Цель

Первая фичевая секция клиентского приложения: **read-only просмотр результатов выполненных тренером тренировок**. Клиент видит, какие тренировки с ним проводил тренер, что фактически делалось (упражнения, подходы — факт против плана) и где достигнут прогресс/рекорд.

Эта секция также задаёт **переиспользуемый паттерн подключения клиентских фичевых роутов** (resolveScope → существующий доменный сервис), на который лягут остальные секции.

## Решения брейншторма (2026-06-03)

- У клиента два типа тренировок: **тренерские** (назначает/проводит тренер) и **свои** (клиент сам ведёт). Эта спека — только про **результаты тренерских**, read-only.
- Тренерскую тренировку «в процессе» клиент НЕ видит — только завершённый результат. Значит клиентский список/деталь показывают **только `status = 'completed'`**.
- Подтверждение/отклонение назначенной тренировки — раздел **Календарь** (отдельно).
- «Свои тренировки» (полный конструктор + выполнение клиентом) — **отдельный следующий под-проект**, не здесь.
- Полные графики динамики — раздел **Статистика** (отдельно). Здесь «прогресс» = факт vs план + бейдж рекорда.

## Объём (v1)

Только просмотр. Никаких write-операций со стороны клиента в этом разделе.

## Бэкенд

### Паттерн интеграции клиентских фич (вводится здесь, переиспользуется далее)

Клиентские фичевые роуты:

1. `requireClient` (cookie `client_sid` → `req.clientAccountId`).
2. `resolveScope(clientAccountId)` → `{ trainerId, clientId } | null`. `null` ⇒ `409 NOT_LINKED`.
3. Вызывают **существующий доменный сервис** с подставленным скоупом. Клиент `clientId`/`trainerId` не передаёт.

`resolveScope` уже возвращается из `registerClientAuthModule` (метод сервиса `client-auth`). Композиционный корень прокидывает его в фичевые модули.

### Новый модуль `apps/api/src/modules/client-app-workouts/`

Тонкий клиентский фасад (по канону routes/module; без своего repo — переиспользует доменный сервис).

- `client-app-workouts.routes.ts` — роуты:
  - `GET /api/client/workouts` → `{ workouts: WorkoutResponse[] }` — только `completed`, сортировка по `completedAt` убыв. Реализация: `resolveScope` → `clientWorkoutsService.list(trainerId, clientId)` → фильтр `status==='completed'` → сортировка.
  - `GET /api/client/workouts/:wid` → `{ workout: WorkoutResponse }` — `clientWorkoutsService.get(trainerId, clientId, wid)`; если тренировка не принадлежит клиенту → доменный сервис уже бросает 404; **дополнительно**: если `status !== 'completed'` → `404 NOT_FOUND` (клиент не видит незавершённые — «во время не видит»).
  - Хелпер `resolveOr409(req)`: достаёт `clientAccountId` (иначе `unauthorized`), зовёт `resolveScope`, при `null` → `AppError(409,'NOT_LINKED',...)`, иначе возвращает `{trainerId, clientId}`.
- `client-app-workouts.module.ts` — `registerClientAppWorkoutsModule(app, { db, clock, resolveScope })`:
  - строит свой экземпляр `makeClientWorkoutsService(makeClientWorkoutsRepo(db), clock)` (сервисы без состояния — дешёвый, принятый в проекте приём),
  - регистрирует роуты под `requireClient`-контекстом (плагин `client-context` уже глобально навешан фундаментом).
- `app.ts`: `const clientAuthSvc = await registerClientAuthModule(...)`; ниже `registerClientAppWorkoutsModule(app, { db: deps.db, clock, resolveScope: (id) => clientAuthSvc.resolveScope(id) })`.

Контракты ответов переиспользуются как есть: `workoutResponseSchema`, `workoutListResponseSchema` из `@trener/shared`. Новых Zod-схем не требуется.

### Изоляция (инварианты)

- Клиент видит только тренировки своего `clientId` (скоуп из сессии; доменный сервис фильтрует по `trainerId`+`clientId`).
- Незавершённые (draft/active/skipped) недоступны клиенту в этом разделе.
- Без привязки тренером → `409 NOT_LINKED`.
- Без сессии → `401`.

## Фронтенд (`apps/web-client`)

### API-слой

- `src/api/workouts.ts`: `useClientWorkouts()` (список завершённых) и `useClientWorkout(wid)` (деталь). Контрактные типы — из `@trener/shared`. Ошибку `NOT_LINKED` обрабатывает гейт приложения (на этот экран клиент попадает только привязанным).

### Экраны

Заменяем заглушку «Тренировки» на:

- **Список** (`/` — вкладка «Тренировки»): завершённые тренировки, **сгруппированы по датам** (заголовки «Сегодня / Вчера / 3 июня»), внутри — карточки: название, время, длительность (если есть), RPE (если есть), число упражнений. Тап → деталь. Пусто → «Пока нет завершённых тренировок». Стиль — токены Acid Flow, как везде.
- **Деталь** (`/workouts/:wid`): шапка (название, дата, длительность, RPE, заметка тренера если есть); список упражнений, по каждому — подходы: **факт** (`повт × кг` или `сек`) и рядом план серым; если факта нет — только план. Бейдж **«рекорд»** на подходе, если это лучший результат по этому упражнению среди загруженных завершённых тренировок (см. ниже). Read-only: ни start, ни отметок, ни «Повторить».

### Бейдж рекорда (ограниченно, клиентская сторона)

Небольшой локальный хелпер `src/lib/records.ts`: по всему загруженному списку завершённых тренировок для каждого упражнения находит максимум (по `actualWeightKg`, при равенстве — по `actualReps`/`actualTimeSec`) и помечает подход, где он достигнут, как рекорд. Чистая функция, юнит-тест. Это не дублирует Статистику (там динамика/графики) — здесь только пометка «лучшее».

### Навигация

Маршрут детали добавляется в основной shell (под нижней навигацией). Вкладка «Тренировки» (`/`) — список; `/workouts/:wid` — деталь с кнопкой «назад» к списку.

## Тестирование

- **Бэкенд** `client-app-workouts.isolation.itest.ts`: клиент A видит только свои завершённые; чужая тренировка → 404; незавершённая своя → 404; без привязки → 409; без сессии → 401; список содержит только `completed`, отсортирован по `completedAt` убыв.
- **Бэкенд** unit/route: фасад делегирует сервису с верным скоупом.
- **Фронт**: юнит `records.ts` (рекорд определяется корректно, граничные случаи — нет факта, равные веса); smoke списка (группировка по датам, пустое состояние) и детали (факт vs план, бейдж).

## Обработка ошибок

- `401` (нет сессии) и `409 NOT_LINKED` (не привязан) — на этот экран привязанный клиент не попадает, но API их возвращает; фронт-гейт уже разруливает (экран входа / «Подключение»).
- `404` — тренировка не найдена/не завершена/не принадлежит клиенту → экран «Тренировка не найдена» с возвратом к списку.

## Вне объёма (явно)

- Любые write-операции клиента над тренировками.
- «Свои тренировки» клиента (конструктор + выполнение) — следующий под-проект.
- Подтверждение/отклонение назначенных тренировок — раздел «Календарь».
- Графики прогресса/динамика — раздел «Статистика».
- Кнопка «Повторить».

## Инварианты, которые НЕ нарушаем

- `repo` остаётся `trainerId`-скоупленным; клиентский фасад подставляет скоуп через `resolveScope`, не обходя сервис/repo.
- `*.routes.ts` не импортирует repo напрямую (фасад-модуль собирает сервис в `*.module.ts`).
- Вход/выход — только Zod-контракты из `@trener/shared`.
