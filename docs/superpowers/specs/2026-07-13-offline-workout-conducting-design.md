# Офлайн-устойчивость тренерского приложения — Фаза 1: проведение тренировки офлайн

**Статус:** дизайн согласован, ждёт вычитки перед планом.
**Дата:** 2026-07-13.

## Цель

Тренер в зале, где часто нет связи, должен спокойно **создать, провести и завершить
тренировку офлайн**; всё сохраняется локально и автоматически синхронизируется при
возврате связи, без потери данных и без дублей.

## Контекст и проблема

Сейчас приложение — тонкий онлайн-клиент. Всё, кроме каталога упражнений и медиа,
идёт вживую на сервер. Проведение тренировки (`ActiveWorkoutScreen._run`) на каждое
действие (отметка/правка/добавление подхода, завершение) шлёт запрос; офлайн он падает,
показывает «Не удалось сохранить», и **действие теряется**. Нет детекта сети, очереди
правок, ретраев.

Уточнённый сценарий (из брейншторма):

- Связи **часто нет с самого начала** (не только обрыв посреди).
- Тренировка обычно **создаётся на месте** (выбор шаблона из базы или сборка с нуля),
  а не берётся заранее подготовленной.
- Долгосрочная цель — офлайн для **всех** правок (замеры, финансы…), но достигаем её
  **поэтапно**: общий движок + первая фаза = проведение тренировки.

## Границы

**В Фазе 1:**

- Общий офлайн-движок (детект сети, кэш чтения, очередь записи, движок синка, индикатор).
- Провод движка в поток проведения тренировки: офлайн выбрать шаблон/собрать → создать →
  провести (отметки, правки значений, добавить/удалить подход, добавить/удалить/переставить
  упражнение) → завершить → синк при связи.
- Кэш чтения для того, что нужно офлайн: клиенты, шаблоны (каталог уже cache-first).
- Бэкенд: идемпотентный эндпоинт импорта проведённой тренировки + миграция.

**Не в Фазе 1 (следующие фазы, тем же движком, отдельными спеками):**

- Офлайн-замеры, финансы, правки клиента и прочие домены.
- Офлайн-**подготовка** тренировки заранее (наша «Создать тренировку → черновик наверх»
  остаётся отдельным онлайн-потоком).
- Офлайн-создание **новых клиентов** (Фаза 1 ссылается только на существующих, закэшированных).
- Офлайн-просмотр остальных экранов (клиенты/календарь/история как read-cache) — может
  войти в фазу read-cache, но не обязателен для проведения.

## Архитектура

Общий движок живёт в `packages/core` (переиспользуется тренером и клиентом). Четыре части:

1. **NetworkStatus** — `isOnlineProvider`. `connectivity_plus` даёт грубый сигнал
   (есть ли сеть), плюс лёгкая проба реального запроса к API (случай «Wi-Fi есть,
   интернета нет» типичен для зала). Экспонирует `isOnline` как провайдер.

2. **Кэш чтения (CachedQuery)** — обобщение нынешнего паттерна каталога
   (`TrainerCatalogNotifier`): запрос отдаёт последнюю сохранённую на диск копию
   мгновенно, в фоне обновляет; офлайн остаётся на копии. Применяем к клиентам и шаблонам.

3. **Outbox (очередь записи)** — персистентная очередь исходящих изменений. Элемент:
   `{ id: клиентский UUID, kind: string, payload: json, status: pending|sending|failed,
attempts: int, createdAt }`. Методы `enqueue / list / markSent / markFailed`.
   Хранится на диске (файловый стор, как `LocalJsonStore`).

4. **SyncEngine** — подписан на `isOnline`. Когда связь есть — сливает очередь **по
   порядку (FIFO)** через зарегистрированные обработчики по `kind`; ретраит сетевые сбои
   с backoff; отдаёт `syncStatusProvider` (`pendingCount`, `syncing`, `lastError`).

**Ключевой принцип — offline-first:** проведение **всегда** идёт через локальный документ
тренировки, а не через прямые вызовы API. Онлайн `SyncEngine` просто сливает очередь сразу.
Поэтому обрыв посреди тренировки «просто работает»: состояние всегда локальное, меняется
только «когда отправилось» — нет переключения режимов online↔offline.

## Модель синка: документ, а не операции

Сервер генерирует id (`deps.newId()`), а гранулярные операции подхода адресуются по
серверным `:wid/:pos/:idx`. Переигрывать мелкие операции офлайн хрупко (позициями управляет
сервер). Поэтому синхронизируем **весь документ тренировки одним вызовом**:

- Офлайн вся тренировка — **локальный документ** (`LocalWorkout`): план + факт (actual) +
  отметки `done` + статус + `startedAt/completedAt` + заметки/RPE + `excludedFromBalance`.
  UI правит документ мгновенно и читает из него.
- Каждой офлайн-созданной тренировке присваивается **клиентский UUID** — он же ключ
  идемпотентности.
- При `complete()` документ кладётся в Outbox как `{ kind: 'workout.import', payload: <документ> }`.
- Обработчик синка шлёт `POST /api/clients/:id/workouts/import` со всем документом и UUID.
  Сервер делает **upsert по idempotency_key**: есть запись с этим UUID — вернуть её (не
  дублировать), иначе создать в финальном виде. Повторная отправка после обрыва безопасна.
- Ссылки на **существующего** клиента используют его реальный (закэшированный) id —
  ремаппинг id в Фазе 1 не нужен.

Общий движок переиспользует ту же идею: любой элемент Outbox — идемпотентный upsert по
клиентскому UUID; тренировки шлют через `import`, будущие домены — через свои идемпотентные
create-эндпоинты (ключ добавляется при подключении каждого домена).

## Изменения на бэкенде (модуль `client-workouts`)

Модуль сейчас **не** трогается параллельной работой (в отличие от `measurements`/`sessions`) —
конфликтов нет.

1. **Эндпоинт** `POST /api/clients/:id/workouts/import` (preHandler `[requireAuth,
requireClientAccess]`), принимает полный документ проведённой тренировки + `idempotencyKey`
   (клиентский UUID). Возвращает `WorkoutResponse`.
2. **Миграция**: колонка `idempotency_key` в таблице тренировок; **частичный уникальный
   индекс** на `idempotency_key WHERE idempotency_key IS NOT NULL` (UUID уникален сам по себе,
   scope до пары не нужен). NULL для обычных (не-импортных) записей.
3. **Сервис импорта**: upsert по ключу; создаёт запись сразу в финальном состоянии
   (упражнения, подходы с planned+actual+done, статус completed, время, заметки, баланс-флаг).
   Запускает **те же побочки, что обычное завершение** (`onCompleted`-хук): списание баланса,
   отметка/создание занятия в календаре, пуш клиенту — **в момент синка** (отложенно; тренер
   видит «отправлено», клиент получает пуш при возврате связи). Согласовано.
4. Валидация: как в обычном create (упражнения видимы тренеру; иначе — понятная ошибка,
   которую движок покажет как «не удалось отправить», не блокируя остальную очередь).

## Провод в тренерское приложение (Фаза 1)

- **`LocalWorkout` + `LocalWorkoutController`** — модель локального документа и контроллер
  (персистится). Операции `create / start / updateSet / addSet / deleteSet / addExercise /
removeExercise / reorder / complete` правят локальный документ и сохраняют его на диск.
  `ActiveWorkoutScreen` перецепляется на контроллер вместо прямых `_api`-вызовов и `_run`.
- **Выбор клиента и шаблона** офлайн: `trainerClientsProvider` и `trainerTemplatesProvider`
  переводятся на cache-first (CachedQuery).
- **Завершение** кладёт документ в Outbox (`workout.import`).
- **Индикатор синка** и **баннер офлайна** — из `syncStatusProvider` / `isOnlineProvider`.

## UX

- **Баннер офлайна** — тонкая полоса «Офлайн — изменения сохранятся и отправятся при связи»,
  когда `isOnline == false`. Нейтральный тон, без красных ошибок.
- **Статус синка** — ненавязчивый индикатор: «⟳ N ждут отправки» → «Синхронизация…» →
  скрыт, когда всё отправлено. Место — шапка или у нижнего меню.
- **Проведение выглядит одинаково** офлайн и онлайн. Завершённая, но не отправленная
  тренировка помечается «не отправлено», пока не синкнется.
- Сохранения офлайн **не показывают** «Не удалось сохранить» — они успешны локально.

## Обработка ошибок и крайние случаи

- **Несколько тренировок подряд** (смена из клиентов) → N импортов в очереди, FIFO.
  Идемпотентность → повторы не дублируют.
- **Сбой отправки**: сетевой → тихий ретрай с backoff; отказ сервера (не сеть, напр.
  невидимое упражнение) → пометить `failed`, показать, **не** блокировать остальные элементы.
- **Приложение убили посреди тренировки** → локальный документ персистится → при открытии
  тренировка **восстанавливается** (resume). Очередь тоже переживает перезапуск.
- **Время** начала/завершения фиксируется по часам телефона в момент проведения и уходит в
  импорт (сервер доверяет времени офлайн-проведённой; возможный сдвиг часов приемлем).
- **Конфликты**: офлайн-созданная тренировка — новая сущность, серверного конфликта нет.
  Правка уже существующих серверных сущностей офлайн — вне Фазы 1.
- **Побочки** (баланс/календарь/пуш) — в момент синка.

## Тестирование (по стандартам репо)

- **Unit (Dart):** `Outbox` (порядок/персист/статусы), `SyncEngine` (слив, ретрай, backoff,
  идемпотентность при повторе), `LocalWorkoutController` (мутации документа + resume из
  персиста), `CachedQuery` (cache-first, офлайн-фолбэк).
- **Backend (vitest):** unit сервиса импорта (upsert, побочки вызваны один раз); repo itest
  против Postgres (двойной импорт с тем же ключом → **одна** запись); isolation-тест
  (тренер A не импортирует в пару тренера B; без auth → 401).
- **Ручной сквозной:** авиарежим → выбрать шаблон/собрать → создать+провести+завершить →
  вернуть связь → проверить, что синкнулось один раз и появилось в истории; убить приложение
  посреди → переоткрыть → resume; несколько тренировок офлайн подряд → все доехали.

## Риски

- **Общий API** используется и вебом, и клиентским приложением; деплой бэкенд-изменений
  авто-раскатывается на прод. Импорт-эндпоинт и миграция должны быть строго аддитивны и не
  ломать существующие потоки.
- **Рефактор `ActiveWorkoutScreen`** на локальный документ — центральная и рискованная часть;
  нужен аккуратный перенос всех действий подхода на контроллер с сохранением текущего UX.
- **Расхождение с «черновик наверх»**: подготовка заранее остаётся онлайн; офлайн-путь —
  только «создать на месте». Пересечение потоков держать явным, чтобы не задвоить сущности.

## Дальнейшие фазы (вне этого спека)

- Фаза 2+: подключить к тому же движку замеры, финансы, правки клиента (каждая — свой
  идемпотентный create + провод контроллера).
- Read-cache остальных экранов (клиенты/календарь/история) для полноценного офлайн-просмотра.
- Быстрый выигрыш рядом: авто-ретрай сетевых ошибок и баннер — уже входят в движок Фазы 1.
