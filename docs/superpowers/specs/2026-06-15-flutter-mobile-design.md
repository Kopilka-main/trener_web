# Мобильные приложения на Flutter (клиент + тренер) — дизайн

**Статус:** одобрено пользователем. Фаза 1 (фундамент + сквозной срез).

## Цель

Нативные приложения iOS + Android для обоих продуктов (клиент и тренер) поверх
существующего API Trener_Prod (Fastify + Postgres). Драйверы: App Store / Play,
нативный UX, доступ к железу, нормальные пуши (APNs/FCM). Веб остаётся параллельно —
общий бэкенд, один аккаунт, синхронные данные.

## Архитектурные решения

- **Online-first** (как веб: всё через API; локальная БД/оффлайн — позже, точечно).
- **Фирменный вид + нативное поведение**: тёмная тема и акцент через токены темы,
  но нативные переходы/жесты/прокрутка.
- **Монорепо Flutter** с общим пакетом `core` + два тонких приложения.
- **Авторизация — переиспользуем существующий сессионный токен**: тот же токен из
  `sessions_auth` / `client_sessions_auth`, но доступен через `Authorization: Bearer`
  и возвращается в теле логина. Cookie продолжает работать для веба. Без новых таблиц.
- Состояние — **Riverpod**; роутинг — **go_router**; сеть — **Dio**; хранение токена —
  **flutter_secure_storage**; пуши — **FCM** (iOS через APNs, Android напрямую).

## Фаза 1 — объём

### 1. Бэк: токен-авторизация (рядом с cookie)

- `tenantContext` (тренер) и клиентский контекст читают токен из cookie ИЛИ из
  `Authorization: Bearer <token>`.
- Логин/регистрация тренера (`/api/auth/*`) и клиента (`/api/client-auth/*`)
  дополнительно возвращают `{ token }` в теле (тот же сессионный токен).
- Logout по токену работает.
- Тесты: itest «логин → Bearer → защищённый роут» для тренера и клиента; веб-cookie
  не сломан.

### 2. Flutter-фундамент (`mobile/`, пакет `core`)

- `core/models` — рукописные Dart-DTO для среза (me/login + данные главной).
- `core/api` — `ApiClient` (Dio): base URL, интерсептор Bearer, 401 → разлогин.
- `core/auth` — secure-storage токена, `AuthController` (Riverpod), login/register/connect.
- `core/theme` — токены цветов/шрифтов, параметризованы под приложение (тренер = лайм,
  клиент = розовый) из текущего CSS.
- `core/push` — инициализация FCM + регистрация device-token на бэке.

### 3. Вертикальный срез (оба приложения)

- **Вход:** старт → токен из secure storage → `/me` → главная; иначе экран входа.
  Логин → `{token}` → сохранить → главная. Клиент: экран «подключиться по коду».
- **Главная:** перенос текущей главной каждого приложения с реальными данными API.
  Плитки → заглушки (следующие фазы).
- **Пуши:** FCM init + регистрация device-token + один тестовый пуш (доказать iOS-пуши).

### 4. Проверка

- Бэк: itest-ы токен-логина (тренер + клиент).
- Flutter: widget-тест входа + unit на токен-интерсептор.
- Приёмка: оба приложения в симуляторе iOS / эмуляторе Android → логин → главная с
  реальными данными → тестовый пуш.

## Ограничения окружения

- Код, бэк и Android-сборку можно делать на Windows.
- iOS-сборка/подпись/публикация требует Mac + Xcode (вне текущей машины).
