# Клиентское приложение — раздел «Профиль», v1

Дата: 2026-06-03
Статус: дизайн на ревью
Ветка: `feat-web-design`

## Цель

Дать клиенту экран **«Профиль»**: отредактировать свои данные (имя, фамилия, дата рождения, контакты, «о себе»/цели), увидеть статус подключения к тренеру и выйти из аккаунта. Профиль клиент заполняет о себе; тренерский просмотр этих данных — отдельный следующий шаг (не в этом v1).

Дополняет уже сделанную модель «тренер опционален»: самостоятельный клиент получает полноценный профиль и место для выхода (логаут переезжает сюда с экрана подключения).

## Решения брейншторма (2026-06-03)

- Редактируемые поля: имя, фамилия, дата рождения, контакты (типизированный список), «о себе»/цели. **Фото — НЕ в этом v1** (подход A, см. ниже).
- На экране также: кнопка **«Выйти»** (переносится сюда; убирается с экрана подключения) и **статус подключения** (привязан / «Подключить» → /connect).
- Email и смена пароля — вне объёма.
- Тренерский просмотр «клиент о себе» — следующий шаг.

## Подход к фото (A — выбран)

Таблица `files` требует `trainerId` (notNull) и хранит путь `<trainerId>/<clientId>/...` — файлы жёстко привязаны к тренеру. У аккаунта клиента тренера может не быть (самостоятельный режим), поэтому аватар клиент-аккаунта не ложится на текущую инфраструктуру. **Фото откладываем** до обобщения файлового хранилища под владельца-аккаунт (отдельный инфра-шаг, делается вместе с тренерским просмотром или раньше). v1 профиля — без фото.

## Модель данных

Расширяем `client_accounts` (миграция drizzle):

- `birthDate text` — дата рождения, строка `YYYY-MM-DD`, NULL = не указана (как в `clients.birthDate`).
- `contacts jsonb` `$type<{type:string; value:string}[]>` NOT NULL default `[]` (как в `clients.contacts`).
- `bio text` — свободный текст «о себе»/цели, NULL = не указан.

`firstName`/`lastName`/`avatarFileId`/`email` уже есть.

## Бэкенд

### Контракты (`@trener/shared`, файл `client-auth.ts`)

- Локальная `contactSchema` `{ type: string(1..40), value: string(1..200) }` (зеркало существующих в `auth.ts`/`clients.ts`; держим client-auth самодостаточным).
- `clientAccountResponseSchema` += `birthDate: string|null`, `contacts: contactSchema[]`, `bio: string|null`. (Влияет на `clientMeResponseSchema` — фронт-гейт/тесты уже используют `account`, добавление полей обратносовместимо.)
- Новый `updateClientAccountRequestSchema`:
  - `firstName` string trim 1..100 optional
  - `lastName` string trim 1..100 optional
  - `birthDate` string `YYYY-MM-DD` nullish (валидация формата как у клиента тренера; пустое → null)
  - `contacts` `contactSchema[]` max 20 optional
  - `bio` string trim max 2000 nullish
- Тип `UpdateClientAccountRequest`.

### client-auth модуль

- Repo: `updateAccount(id, patch)` (Drizzle update + returning), как `updateTrainer` в auth.repo. `findAccountById`/`createAccount` обновляются под новые поля (возврат включает birthDate/contacts/bio).
- Сервис: `updateMe(clientAccountId, input)` — собирает patch только из определённых полей (`exactOptionalPropertyTypes`), `birthDate`/`bio` `null` = очистка; возвращает `ClientAccountResponse`. `toAccountResponse` дополняется новыми полями.
- Роут: `PATCH /api/client/auth/me` (`requireClient`-контекст; внутри `if (!req.clientAccountId) 401`), body `updateClientAccountRequestSchema`, ответ `{ account }`.

### Изоляция/ошибки

- Без сессии → 401.
- Клиент правит ТОЛЬКО свой аккаунт (id из сессии, не из тела).
- Валидация формата даты/длин — Zod; некорректное → 400.

## Фронтенд (`apps/web-client`)

### API

- `src/api/profile.ts` (или дополнить `api/auth.ts`): `useUpdateClientProfile()` — мутация PATCH `/client/auth/me`, при успехе инвалидирует `clientMeQueryKey`. Текущие значения берём из `useClientMe()`.

### Экран «Профиль» (вкладка `/profile`, заменяет заглушку)

- Форма (значения из `me.account`): имя, фамилия; дата рождения (`<input type="date">`, color-scheme: dark — как у тренера); контакты (типизированный список с чипами типов `Телефон/WhatsApp/Telegram/MAX/Instagram/Прочее`, добавить/удалить — зеркало тренерского UI, но компонент свой в web-client); «о себе»/цели (textarea). Кнопка «Сохранить» (disabled при отсутствии изменений/в процессе), ошибка валидации/сети — нейтральным `ink`-текстом.
- **Статус подключения**: привязан (`me.link !== null`) → строка «Вы подключены к тренеру»; не привязан → кнопка/ссылка «Подключить тренера» → `/connect`.
- **«Выйти»** внизу (вызывает `useClientLogout`).

### Правка экрана подключения

- Убрать кнопку «Выйти» с `ConnectPage` (логаут теперь в Профиле). Оставить «Продолжить».

## Тестирование

- **Бэкенд**: unit сервиса `updateMe` (patch только определённых полей; birthDate/bio null = очистка); isolation/route itest — PATCH меняет свой аккаунт, без сессии → 401, `me` после PATCH отдаёт новые значения; repo itest на `updateAccount`.
- **Фронт**: smoke профиля — значения из мокнутого `me`, «Сохранить» зовёт мутацию с собранным payload; статус подключения (привязан/не привязан) рендерится; «Выйти» присутствует. Обновить `ConnectPage`/App-тесты под перенос логаута.

## Вне объёма (явно)

- Фото профиля (подход B — обобщение `files`).
- Тренерский просмотр «клиент о себе».
- Смена пароля, смена/верификация email.
- Свои тренировки клиента, календарь, чат, статистика — отдельные под-проекты.

## Инварианты

- Клиент правит только свой `client_accounts` (id из сессии).
- `*.routes.ts` не импортирует repo (сборка в `*.module.ts`).
- Вход/выход — только Zod-контракты `@trener/shared`.
