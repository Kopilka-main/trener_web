# Переработка экрана проведения тренировки: блоки упражнений с подходами

Дата: 2026-07-10
Область: тренерское приложение (`mobile/apps/trainer`) + бэкенд (`apps/api`)

## Проблема

На экране проведения тренировки одинаковые упражнения показываются отдельными
карточками (нумеруются «Название 1/2/3»), потому что подход сейчас добавляется
дублированием всего упражнения (нет API «добавить один подход»). Подходы всегда
раскрыты, редактирование/удаление разбросаны по мелким кнопкам. Нужен компактный
экран: один блок на упражнение со свёрнутыми подходами и свайп-действиями.

## Согласованные решения

- **`+1`** — всегда молча добавляет копию подхода (те же плановые параметры). Без
  удержания. Менять параметры — отдельным действием «редактировать».
- **Группировка** — соседние `WorkoutExercise` с одинаковым `exerciseId`
  склеиваются в один блок; их подходы идут подряд.
- **Раскрытие** — блок свёрнут по умолчанию, разворачивается по стрелке.

## Архитектура

### Бэкенд: два эндпойнта для одного подхода

Подходы хранятся строками в `client_workout_sets` (`exercise_position`,
`set_index`). Добавляем:

- `POST /api/clients/:id/workouts/:wid/exercises/:pos/sets`
  Тело: плановые параметры (`plannedReps/WeightKg/TimeSec/RestSec`, все опц.).
  Вставляет строку со следующим `set_index`. Возвращает обновлённую тренировку.
- `DELETE /api/clients/:id/workouts/:wid/exercises/:pos/sets/:idx`
  Удаляет подход, переиндексирует оставшиеся `set_index` (0..n-1). Если это был
  **последний** подход упражнения — удаляет и само упражнение (как `removeExercise`).
  Возвращает обновлённую тренировку.

Слои: `client-workouts.repo.ts` (методы `addSet`, `deleteSet`),
`client-workouts.service.ts`, `client-workouts.routes.ts`, Zod-схемы в
`packages/shared` (`addWorkoutSetRequestSchema`; переиспользуем `setParams`).
Юнит-тесты сервиса на оба метода (включая удаление последнего подхода → удаление
упражнения, и переиндексацию).

`updateSet` (существующий) переиспользуется для «редактировать».

### Фронтенд тренера

**API** (`trainer_workouts.dart`): `addSet(clientId, wid, pos, WorkoutSet template)`
и `deleteSet(clientId, wid, pos, idx)`. Метод `duplicateExercise` больше не нужен
на экране (удаляется его вызов; сам метод можно оставить или убрать).

**Экран** (`active_workout_screen.dart`):

- **Группировка**: перед рендером `pending`/`completed` соседние упражнения с
  одинаковым `exerciseId` сворачиваются в модель-группу
  `_ExGroup { exerciseId, title, positions: List<int>, sets: List<(pos, WorkoutSet)> }`.
  Подходы группы нумеруются сквозно.
- **`_ExerciseBlock`** (замена `_ActiveExerciseCard`): свёрнутая шапка
  `[drag] Название · «N подходов · выполнено X/N» · [▸]`; тап по шапке/стрелке
  разворачивает список подходов. Состояние раскрытия — `Set<String>` по
  `exerciseId` (по умолчанию пусто = свёрнуто). Реордер — целыми блоками
  (по первой позиции группы).
- **`_SwipeSetRow`** (замена `_activeSetRow`): строка подхода
  `№ · метрики · ✓(тап = отметить выполнен, хаптик)`; **свайп влево** (через
  `flutter_slidable`) открывает `[+1] [редактировать] [удалить]`:
  - `+1` → `addSet(pos, копия параметров)` + хаптик;
  - `редактировать` → существующий `_SetEditor` + `updateSet`;
  - `удалить` → `confirmDelete('Удалить подход?')` → `deleteSet(pos, idx)`.
- Завершённые блоки (все подходы `done`) уходят в секцию «ЗАВЕРШЕНО» как сейчас.
- Шапка с таймером/«Готово» и демо «Следующее упражнение» — без изменений.

**Зависимость**: подключить `flutter_slidable` в `apps/trainer/pubspec.yaml`.
Свайп — на подходах внутри блока, драг — на блоках; конфликта с
`ReorderableListView` нет.

### Порядок действий с индексами (важно)

`deleteSet` меняет `set_index` остальных подходов упражнения. Экран после каждого
мутирующего вызова получает свежую тренировку с сервера (как сейчас через `_run`),
поэтому индексы всегда актуальны — оптимистичных правок не делаем.

## Область (scope)

- Только тренерское приложение. Клиентское приложение — паритетом отдельным
  заходом (по правилу «клиент = тренер один-в-один»), в этот спек не входит.
