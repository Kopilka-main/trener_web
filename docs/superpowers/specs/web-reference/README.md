# Эталонный справочник веба → инструкция для мобайла

> **Источник истины — веб** (`apps/web` тренер, `apps/web-client` клиент). Это
> зрелый прод. Мобильные приложения (`mobile/apps/trainer`, `mobile/apps/client`)
> строятся и сверяются строго по этому справочнику. Любое расхождение мобайла —
> баг мобайла, пока не доказано обратное (явное продуктовое решение).

## Зачем

Прошлые сверки были **код-в-код (статика)** — ловили наличие экрана, но не
рантайм-поведение. Этот справочник фиксирует, ЧТО именно должен делать каждый
экран: дизайн, данные, действия, вызовы API, состояния, навигация. По нему
мобайл реализуется и проверяется в рантайме.

## Структура

- `00-design-system.md` — токены (светлая/тёмная), типографика, отступы,
  паттерны (плитка/карточка/чип/кнопка/FAB), инвентарь общих компонентов.
- `client/<screen>.md` — по одному файлу на экран клиента.
- `trainer/<screen>.md` — по одному файлу на экран тренера.
- `coverage.md` — матрица «веб-экран → мобильный экран → рантайм-статус».

## Шаблон описания экрана

Каждый файл экрана заполняется по разделам:

```
# <Экран> — <web-page.tsx>

**Маршрут:** <path> · **Точки входа:** откуда переходят
**Назначение:** одна фраза

## Макет (сверху вниз)
Блоки в порядке отображения, с указанием компонентов ДС и токенов.

## Данные
Каждое отображаемое поле → источник (хук/endpoint, поле ответа), формат.

## Действия
Каждый жест/кнопка → эффект → API (МЕТОД путь, тело → ответ) → инвалидации.

## Состояния
loading / empty / error / ветки по роли/праву/условию (что показывается).

## Навигация
Каждый тап → маршрут.

## Бизнес-правила и edge-cases
Нетривиальная логика, формулы, граничные случаи.

## Сводка эндпоинтов
Список всех endpoint'ов экрана (метод + путь + назначение).
```

## Конвенции

- Пути API — как в коде (`/api/...`), плейсхолдеры `:id`.
- Цвета/шрифты — токенами ДС (`accent`, `ink`, `font-display`…), не хексами.
- «акцент» = `--color-accent` (светлая #f72585 / тёмная #d4ff3d). На экране —
  максимум один акцентный fill (правило «один acid-fill»).
