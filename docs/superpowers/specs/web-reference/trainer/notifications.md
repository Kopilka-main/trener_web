# Уведомления (тренер) — NotificationsPage.tsx (+ lib/notifications.ts)

**Маршрут:** `/notifications` · **Точки входа:** главная тренера (плитка «Уведомления»)
**Назначение:** производные уведомления тренера, СОБРАННЫЕ НА КЛИЕНТЕ из списков
клиентов, занятий и балансов. Две вкладки: «События» и «Требует действия».
Отдельного endpoint уведомлений НЕТ — всё считает `buildNotifications`.

## Макет (сверху вниз)

1. **ScreenHeader** — «Уведомления», назад → `/`, справа **HoldToDelete**
   (корзина): очистить (скрыть) все видимые элементы текущей вкладки. Показывается
   только если есть что чистить.
2. **Табы** (`bg-chip`): «События» / «Требует действия» (+ красный бейдж-счётчик
   открытых алертов на втором).
3. **Список** текущей вкладки:
   - **EventCard** — иконка по виду (Cake коралл / CheckCircle2 acid / CalendarCheck
     acid / CalendarPlus ink) + заголовок (имя) + время справа + сообщение. Тап →
     карточка клиента (если есть `clientId`). X — скрыть.
   - **AlertCard** — иконка по типу (Wifi/Dumbbell/CalendarOff/CalendarX2/AlertTriangle),
     цвет danger или amber по severity, mono-кикер `headline` + имя клиента +
     сообщение + «Открыть карточку ›». Тап → клиент. X — скрыть.

## Данные / на чём строятся события

Источники (всё чтение):
| Хук | Endpoint | Назначение |
| -------------------- | -------- | ---------- |
| `useClients()` | `GET /clients` | клиенты (имена, дни рождения) |
| `useSessions(from,to)` | `GET /sessions?from&to` | занятия за период **[−14дн … +30дн]** |
| `usePackageBalances()` | `GET /packages/balances` | остатки оплаченных тренировок по клиентам |

`paidClientIds` = клиенты с `remaining > 0` (передаются в builder для алерта «нет
занятий на неделю» — шлём только оплатившим).

### События (вкладка «События», сорт по `ts` убыв.)

| kind        | Условие                                                                    | Сообщение                                  |
| ----------- | -------------------------------------------------------------------------- | ------------------------------------------ |
| `completed` | `status==='completed'` и дата в `[−14дн … сегодня]`                        | «Проведено занятие · <title>»              |
| `confirmed` | `planned` + `clientConfirmation==='confirmed'`, дата в `[сегодня … +14дн]` | «Клиент подтвердил занятие · <title>»      |
| `planned`   | `planned` (не confirmed/declined), дата в `[сегодня … +14дн]`              | «Запланировано занятие · <title>»          |
| `birthday`  | `c.birthDate` совпадает по дню+месяцу с сегодня                            | «Сегодня день рождения 🎂», when=«сегодня» |

### Требует действия (вкладка «action»)

| type           | severity | Условие                                                                | Сообщение                                                |
| -------------- | -------- | ---------------------------------------------------------------------- | -------------------------------------------------------- |
| `cancelled`    | danger   | `status==='cancelled'`, дата в `[−14 … +14]`                           | «<дата> — переназначьте или свяжитесь с клиентом»        |
| `declined`     | danger   | не cancelled + `clientConfirmation==='declined'`, дата в `[−14 … +14]` | «<дата> — согласуйте другое время»                       |
| `online_today` | warn     | `planned` + `isOnline` + дата === сегодня                              | «Сегодня в <time> · <title>»                             |
| `no_upcoming`  | warn     | клиент из `paidClientIds` БЕЗ planned-занятий в `[сегодня … +7дн]`     | «Оплачены тренировки, но нет записи на ближайшие 7 дней» |

`labelDate(iso,time)` → «3 июн, 14:30» / «3 июн». `ts` для сортировки — `Date.parse(date+time)`;
ДР получает `Date.now()+1` (наверх).

## Действия

| Жест                       | Эффект                                                             | API                                                       |
| -------------------------- | ------------------------------------------------------------------ | --------------------------------------------------------- |
| Тап по карточке            | Перейти к клиенту (если `clientId`)                                | → `/clients/:clientId` (клиентский navigate)              |
| X на карточке              | `dismiss(id)` — добавить id в `dismissed`                          | localStorage `notifications_dismissed`                    |
| Удержание корзины (header) | `clearAll()` — скрыть все видимые id текущей вкладки               | тот же `dismissed`                                        |
| Заход на экран             | пометить текущие алерты «просмотренными» → бейдж на главной гаснет | localStorage `notifications_seen` (после загрузки данных) |
| Смена таба                 | локальный стейт                                                    | —                                                         |

Мутаций на сервер нет — все эффекты локальные (localStorage + клиентская фильтрация).

## Состояния

- **loading** (`clients.isPending || sessions.isPending`): «Загрузка…».
- **events empty:** «Пока нет событий.».
- **action empty:** «Нет открытых задач — всё в порядке.».
- **dismissed:** скрытые id не показываются (`visibleEvents`/`visibleAlerts` =
  отфильтрованы по `dismissed`).
- Бейдж на табе «Требует действия» = число `visibleAlerts` (красный, danger).
- Корзина-«очистить» видна только при `visibleIds.length > 0`.

## Навигация

назад → `/` · карточка → `/clients/:clientId`.

## Бизнес-правила и edge-cases

- **dismissed** (`notifications_seen` ≠ `notifications_dismissed`):
  - `dismissed` — скрытые пользователем карточки (X / «очистить»), сохраняются.
  - `seen` — «просмотренные» при заходе на экран; используется ТОЛЬКО для счётчика
    на плитке главной (новые id с другими ключами снова поднимут бейдж).
- `no_upcoming` шлётся только при `paidClientIds?.has(c.id)` (оплатил, но не
  записан); если балансы не загружены (`undefined`) — алерт не формируется.
- Стабильные id: `ev:done:<sid>`, `ev:confirm:<sid>`, `ev:plan:<sid>`,
  `ev:bday:<cid>`, `al:cancel:<sid>`, `al:declined:<sid>`, `al:online:<sid>`,
  `al:noup:<cid>` — переживают рефетч (dismiss/seen работают между сессиями).
- Окна дат: события completed [−14…0], planned/confirmed [0…+14], алерты
  cancelled/declined [−14…+14], online — только сегодня, no_upcoming — [0…+7].
- Один acid-fill отсутствует: иконки severity используют danger/amber/coral/accent-text
  по смыслу (это иконки статуса, не действие — соответствует правилу «красный только
  для иконок severity»).

## Сводка эндпоинтов

- `GET /clients` — клиенты (имена, ДР).
- `GET /sessions?from&to` — занятия [−14дн … +30дн].
- `GET /packages/balances` — остатки оплаченных (фильтр `remaining>0` → `paidClientIds`).
- (уведомлений как ресурса на сервере НЕТ — всё считается на клиенте.)
