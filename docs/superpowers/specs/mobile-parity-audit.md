# Аудит паритета: веб ↔ мобильные приложения (Flutter)

> **Источник истины — веб** (`apps/web-client`, `apps/web`). Это зрелый прод.
> Любое расхождение мобайла с вебом считается багом мобайла, пока не доказано
> обратное (явное продуктовое решение). Сверяемся ПО ИСХОДНИКАМ, не по памяти.

## Метод (на каждый экран)

Извлекаем из веб-страницы структурированный список возможностей и сверяем с
мобильным экраном. Каждый пункт получает статус:

- ✅ есть · ⚠️ частично · ❌ нет · ➖ неприменимо на мобайле

Категории проверки:

1. **Данные** — каждое отображаемое поле.
2. **Действия** — каждая кнопка/жест → endpoint + метод + payload.
3. **Состояния** — loading / empty / error / ветки по роли/праву/условию.
4. **Бизнес-правила и edge-cases** — нетривиальная логика.
5. **Навигация** — куда ведёт каждый тап.

Исполнение: на каждый экран — отдельный субагент, читает веб-страницу + мобильный
экран, возвращает таблицу расхождений. Результаты сводятся сюда.

## Шаблон по-экранного разбора

```
### <Экран> — <web-page.tsx> ↔ <mobile_screen.dart>

Данные:
- [ ] поле X — ✅/⚠️/❌ (заметка)
Действия:
- [ ] кнопка Y → POST /api/... — ✅/⚠️/❌
Состояния:
- [ ] empty/error/loading — ✅/⚠️/❌
Бизнес-правила:
- [ ] правило Z — ✅/⚠️/❌
Навигация:
- [ ] тап → /route — ✅/⚠️/❌

ИТОГ: N расхождений. Приоритет: P0/P1/P2.
```
