# Клиентское приложение — фундамент (auth, скоуп, скаффолд)

Дата: 2026-06-03
Статус: дизайн на ревью
Ветка: `feat-web-client`

## Цель

Отдельный фронт для клиентов (`apps/web-client`) поверх **того же** API, что и тренерский.
Клиент сам регистрируется, тренер привязывает его к своей записи клиента по ID, после чего
клиент видит и взаимодействует со своими данными. Бэк один, фронта два, деплой раздельный.

Эта спека описывает **только фундамент** — спину, на которую вешаются фичевые секции.
Сами секции (тренировки, календарь, чат, прогресс, оплата+профиль) — отдельные спеки после.

## Декомпозиция (общая дорожная карта)

1. **Фундамент (эта спека).** Клиентский auth (таблицы + модуль + middleware), резолвер скоупа
   `clientAccount → {trainerId, clientId}`, онбординг/привязка, скаффолд `apps/web-client`,
   общий theme, docker-образ. Внутри приложения — нижняя навигация с пустыми секциями-заглушками.
2. **Секция «Тренировки»** — клиент смотрит назначенные тренировки и отмечает выполнение.
3. **Секция «Календарь»** — клиент смотрит расписание, подтверждает/отменяет своё занятие.
4. **Секция «Чат»** — читает и пишет сообщения тренеру.
5. **Секция «Прогресс»** — смотрит и вносит замеры и фото прогресса.
6. **Секция «Оплата + Профиль»** — смотрит баланс/занятия; редактирует свой профиль.

Каждая секция = свой spec → plan → реализация, по одной, с ревью между.

## Решения, зафиксированные в брейншторме

- **Вход:** саморегистрация email + пароль (как у тренера). Клиент получает свой ID, передаёт
  тренеру; тренер привязывает через существующий флоу «Подключить» в карточке клиента.
- **Один тренер в v1.** Модель данных уже M:N (`trainer_clients`), но UI и резолвер v1 берут
  одну активную связь.
- **Размещение:** новый workspace `apps/web-client` в монорепо. Раздельный docker-образ.
- **Подход к auth (выбран A):** отдельный клиентский auth + резолвер скоупа поверх существующих
  сервисов. Repo остаются `trainerId`-скоупленными — инвариант CLAUDE.md цел.

## Модель данных

Новые таблицы (миграция drizzle, зеркало `trainers`/`sessions_auth`):

- `client_accounts`
  - `id` text PK (случайный, как у тренера) — он же «код подключения», который клиент передаёт тренеру.
  - `email` text NOT NULL, уникальный индекс (нормализованный lowercase+trim через Zod-контракт).
  - `passwordHash` text NOT NULL.
  - `firstName`, `lastName` text NOT NULL.
  - `avatarFileId` text NULL → `files.id` (`on delete set null`) — фото профиля клиента (для секции профиля; в фундаменте колонка заводится, UI позже).
  - `createdAt` timestamptz NOT NULL default now().
- `client_sessions_auth`
  - `id` text PK (токен сессии).
  - `clientAccountId` text NOT NULL → `client_accounts.id` (`on delete cascade`).
  - `expiresAt` timestamptz NOT NULL.
  - `createdAt` timestamptz NOT NULL default now().

Существующая `clients.accountId` обретает смысл: хранит `client_accounts.id` привязанного аккаунта.
FK не добавляем (связь логическая, ставится контролируемо при подключении), но **при подключении
валидируем существование** `client_accounts` с таким id.

## Бэкенд

### Модуль `apps/api/src/modules/client-auth/`

По канону (routes/service/repo/schema), зеркало `auth`:

- `POST /api/client/auth/register` — body `{email, password, firstName, lastName}`; создаёт
  `client_accounts`, ставит сессию (cookie), возвращает `{ account }` (без passwordHash). 409 `EMAIL_TAKEN`.
- `POST /api/client/auth/login` — 401 при неверных данных.
- `POST /api/client/auth/logout` — чистит сессию и cookie.
- `GET  /api/client/auth/me` — `{ account, link }`, где `link = { trainerId, clientId } | null`
  (результат резолвера; null = ещё не привязан).

Сервис переиспользует `hashPassword`/`verifyPassword` из `../../auth/password.js` и паттерн TTL-сессии
из `auth.service.ts` (30 дней). Лимитирование как у тренерского auth (`rateLimit` max 20/мин).

### Плагин `client-context` (отдельная cookie)

Зеркало `tenant-context.ts`, но:

- cookie-имя **`client_sid`** (тренерское — `sid`, не пересекаются даже в одном браузере).
- декорирует `req.clientAccountId?: string` через `onRequest` (читает токен → `findClientSession` →
  проверка `expiresAt`).
- экспортирует guard `requireClient` (нет `clientAccountId` → 401 `Требуется вход`).

Регистрируется в `app.ts` рядом с `tenantContext`. Оба плагина сосуществуют независимо.

### Резолвер скоупа

Чистая функция в сервисном слое: `resolveClientScope(clientAccountId) → { trainerId, clientId } | null`.

- Находит `clients` где `accountId = clientAccountId`.
- Находит активную связь в `trainer_clients` (`status = 'active'`) для этого клиента; v1 — берём
  одну, детерминированно (по `createdAt`, затем `trainerId`). Документируем v1-допущение в коде.
- Нет записи/связи → `null`.

Фичевые клиентские роуты (в секционных спеках) будут: `requireClient` → `resolveClientScope` →
если `null` ⇒ 409 `NOT_LINKED`; иначе вызывают **существующие** сервисы с `{trainerId, clientId}`.
Клиент **никогда** не передаёт `clientId` в теле/параметрах — он берётся из сессии.

### Привязка клиента тренером (доработка `clients`-модуля)

Существующий PATCH клиента (через который тренер ставит `accountId`) дополняется валидацией: если
в патче приходит непустой `accountId` — проверить, что `client_accounts` с таким id существует,
иначе 422 `CLIENT_ACCOUNT_NOT_FOUND`. Пустая строка/null = отвязка (разрешена). Тренерский фронт
(«Подключить» вручную или скан QR через уже добавленный `QrScanner`) шлёт этот же id.

## Фронтенд `apps/web-client`

Скаффолд по образцу `apps/web` (Vite + React 18 + Tailwind v4 + TanStack Query + react-router 6):

- `api/client.ts` — свой `apiFetch` (тот же контракт ошибок `ApiError`, `credentials: 'include'`),
  бьёт в `/api/client/*`. Cookie клиентская (ставит сервер), фронт её не трогает.
- `api/auth.ts` — `useClientRegister/useClientLogin/useClientLogout/useClientMe`.
- **Гейт доступа** (в корневом роутере):
  - не залогинен (`me` → 401) → экраны `/login`, `/register`;
  - залогинен, но `link === null` → экран «Подключение»: показывает свой ID (текст + QR) с подписью
    «передай тренеру, чтобы он подключил тебя»; поллинг/refetch `me` для перехода дальше;
  - залогинен и привязан → основное приложение с нижней навигацией.
- **Нижняя навигация** (one-handed UX, по памяти про bottom controls): Тренировки / Календарь /
  Чат / Прогресс / Профиль. В фундаменте секции — заглушки `StubPage` («Скоро»), наполняются в
  секционных спеках.
- Контракты — из `@trener/shared` (переиспользуем как есть). Клиентские response-схемы для
  `account`/`link` добавляем в `@trener/shared` (новый файл `client-auth.ts`).

### Общий theme

Токены Acid Flow выносятся в `packages/theme/theme.css` (CSS `@theme`-слой Tailwind v4). Импортируют
оба фронта (`apps/web` и `apps/web-client`) — чтобы тема не разъезжалась. `apps/web/src/index.css`
переключается на импорт из общего пакета (изменение тренерского фронта минимальное и визуально
нейтральное). Компоненты клиентский фронт делает свои (UX отличается); общую UI-библиотеку не
выделяем — преждевременно.

## Docker / деплой

В `docker-compose.yml` добавляется второй nginx-сервис `web-client`:

- свой образ из `apps/web-client/dist`, конфиг nginx как у `web` (index.html no-cache, `/assets`
  immutable), свой порт/домен (тренеры — напр. `app.<домен>`, клиенты — `my.<домен>`).
- API-контейнер один на оба фронта. CORS/же-origin: фронты на поддоменах, cookie клиентская
  (`client_sid`) ставится сервером с `sameSite: 'lax'`; конфигурацию доменов и при необходимости
  CORS уточняем при сборке docker-стадии (не в этой спеке — здесь только структура).

Поднимать можно вместе или по отдельности (`docker compose up web-client`).

## Обработка ошибок (коды)

- `EMAIL_TAKEN` (409) — email уже зарегистрирован; фронт показывает у поля email.
- 401 при неверном входе → «Неверный email или пароль».
- `NOT_LINKED` (409) — обращение к фиче без привязки; фронт ведёт на экран «Подключение».
- `CLIENT_ACCOUNT_NOT_FOUND` (422) — тренер вводит несуществующий код подключения.

## Вне объёма (явно)

- Самостоятельная запись на занятия (только просмотр + подтверждение/отмена — в секции «Календарь»).
- Несколько тренеров у клиента (модель готова, UI/резолвер v1 — один).
- Восстановление пароля, верификация email, magic-link, SMS/OTP.
- Push-уведомления.
- Наполнение секций — отдельные спеки.

## Инварианты, которые НЕ нарушаем

- `repo` остаётся скоупленным по `trainerId` (клиентские роуты подставляют скоуп через резолвер,
  не обходят repo).
- Никаких секретов в коде; вход/выход только через Zod-контракты `@trener/shared`.
- `routes` не импортирует `repo` напрямую; слои client-auth следуют тому же ESLint-канону.
